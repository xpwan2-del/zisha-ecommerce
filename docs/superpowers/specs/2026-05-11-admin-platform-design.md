# Zisha Admin 后台管理平台 UI/UX 升级设计

## 目标

将库存中心、库存流水、盘点管理、预警管理和商品管理统一升级为企业级后台体验：专业、数据密集、操作高效、反馈明确，并让库存模块形成“中心页 + 专业子页”的可追溯闭环。

## 已确认方案

采用方案 A：**库存中心首页 + 3 个专业子页**。

- 库存中心首页：作为库存运营驾驶舱，负责总览、风险、待办、最近动态和快捷入口。
- 库存流水页：作为库存变更追溯中心，负责查清楚库存为什么变化。
- 盘点管理页：作为盘点任务推进中心，负责创建、执行、完成和生成流水。
- 预警管理页：作为异常处理中心，负责发现、处理、反馈和留痕。
- 商品管理页：升级为商品资料、价格、库存、分类、促销状态的专业数据管理台。

## 视觉方向

基于 `ui-ux-pro-max` 生成的设计系统，采用 Data-Dense Dashboard 风格：

- 主色：深蓝 `#1E40AF`
- 背景：浅灰蓝 `#F8FAFC`
- 强调色：琥珀 `#F59E0B`
- 表格：高密度、轻边框、清晰状态标签
- 卡片：简洁白底、细边框、弱阴影
- 交互：悬浮高亮、明确焦点、统一成功/错误反馈

## 库存中心首页

首页目标是让管理员快速回答三个问题：

1. 现在库存是否健康？
2. 哪些商品有风险？
3. 下一步应该处理什么？

首页包含：

- KPI 区：SKU 总数、总库存、低库存、缺货、待处理预警、进行中盘点。
- 风险区：优先展示缺货，其次展示低库存，提供查看流水、调库、创建盘点、处理预警动作。
- 待办区：展示待解决预警、未完成盘点、异常任务。
- 最近动态区：展示最近库存流水、最近盘点完成、最近预警解决。
- 快捷操作区：新建盘点、手动调库、查看预警、查看流水。

## 库存流水页

流水页专注追溯，统一筛选条件包括：

- 商品名称 / 商品 ID
- 流水类型
- 操作人
- 时间范围

表格列包括：商品、变更类型、变更前库存、变更数量、变更后库存、来源模块、关联单号、操作人、时间、备注、操作。

动作包括：查看商品、查看关联盘点、查看关联预警、查看订单来源。

## 盘点管理页

盘点页专注任务推进：

- 列表：盘点单编号、状态、商品数量、已录入数量、差异数量、创建时间、完成时间、操作。
- 创建：按商品名、分类、库存状态筛选，支持批量选择。
- 详情：展示账面库存、实盘库存、差异、保存状态。
- 完成：完成后展示调整商品数、盘盈数、盘亏数、生成流水数，并引导查看关联流水。

## 预警管理页

预警页专注异常处理：

- 筛选：全部、待处理、已解决、缺货、低库存。
- 列表：预警类型、商品、当前库存、阈值、状态、触发时间、解决人、解决时间、解决备注、操作。
- 动作：解决预警、查看流水、手动调库、创建盘点。
- 反馈：解决后明确提示，并推荐查看流水或返回库存中心。

## 商品管理页

商品管理页改造为专业数据管理台：

- 顶部指标：商品总数、上架商品、下架商品、缺货商品、低库存商品。
- 筛选条：商品名称、分类、价格区间、库存状态、上下架状态。
- 商品表格：商品图、名称、分类、售价、库存、库存状态、促销状态、更新时间、操作。
- 详情编辑：推荐右侧抽屉或页面内详情面板，避免跳走，保留筛选和分页状态。

## 统一 UI 规范

所有 admin 页面统一：

- 页面壳：标题、描述、右侧动作、指标卡、主内容区。
- 状态标签：正常绿色、低库存黄色、缺货红色、处理中蓝色、完成灰/绿。
- 空状态：标题 + 描述 + 可选动作，不再只显示“暂无数据”。
- 错误状态：统一错误提示和重试按钮，不再只 `console.error` 或 `alert`。
- 成功反馈：所有写操作必须有成功提示，并尽量提供下一步动作。
- 表格：统一表头、行高、hover、空态、loading、操作列。

## 实施顺序

1. 抽取或复用统一 admin UI 组件。
2. 重构库存中心首页。
3. 统一库存流水、盘点、预警子页。
4. 重构商品管理页。
5. 运行 lint/build 验证。

## 风险控制

- 不修改数据库结构。
- 不改变库存核心后端逻辑。
- 不绕过现有库存事务和审计逻辑。
- 所有涉及库存变动的操作继续依赖后端接口完成。
- 前端只做体验升级和页面组织优化。
