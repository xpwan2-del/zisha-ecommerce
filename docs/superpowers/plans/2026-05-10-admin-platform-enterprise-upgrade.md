# 后台管理系统大厂化全面升级 实现计划

> **面向 AI 代理的工作者：** 必需子技能：使用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans 逐任务实现此计划。步骤使用复选框（`- [ ]`）语法跟踪进度。

**目标：** 在不改动前台用户侧模块的前提下，将现有后台升级为覆盖订单、支付、商品、库存、营销、评价、账户、风格、数据看板和系统配置的完整运营管理系统，并以大厂后台标准统一交互、权限、审计、数据口径和风险控制。

**核心约束：**
- 不主动改动前台页面：`src/app/(front)` 当前未使用分组，因此实际约束为不改动 `/cart`、`/products`、`/orders`、`/account`、`/quick-order`、`/payment-result` 等用户侧页面。
- 后台改造范围优先限定在 `src/app/admin/**`、`src/app/api/admin/**`、后台专用组件和后台专用服务。
- 支付、订单、库存、优惠券属于高风险链路，必须先审计现有逻辑，再做最小闭环改造。
- 数据库写入类操作必须提前说明影响表和数据，得到确认后才能执行。
- 前端支付金额永远不能作为可信来源，支付中心只能读取 DB 中的订单金额与支付状态。

**技术栈：** Next.js App Router、React、TypeScript、SQLite/SQL.js、现有 Admin API、现有支付 Strategy + Adapter 架构、现有主题与多语言配置体系。

## 三、实施策略

### 推荐路线

优先选择「后台基础框架 → 订单中心 → 库存中心 → 商品中心 → 支付中心 → 营销中心 → 评价/账户/风格/看板补齐」的顺序。

原因：
1. 后台框架先统一，后续所有页面可以复用同一套表格、筛选、状态和操作规范。
2. 订单、库存、支付是高风险链路，必须在统一日志和操作规范具备后再动。
3. 商品上架依赖分类、库存、图片、价格等基础能力，不能孤立改。
4. 支付中心最重要，但也最危险，应在订单状态和审计日志清楚后改。

### 分阶段目标

| 阶段 | 名称 | 目标 | 风险等级 |
|------|------|------|----------|
| P0 | 后台基础规范 | 统一后台框架、导航、表格、筛选、状态、操作反馈 | 低 |
| P1 | 订单中心 | 完整订单列表、详情、状态扭转、发货、退款审批 | 高 |
| P2 | 库存中心 | 库存总览、流水、调库、盘点、预警、字典联动 | 高 |
| P3 | 商品中心 | 商品维护、大厂级上架校验、草稿和上下架流程 | 中高 |
| P4 | 支付中心 | 通道配置、支付订单、退款订单、回调日志、异常处理 | 高 |
| P5 | 营销中心 | 优惠券、促销、活动和营销效果分析 | 中 |
| P6 | 运营补齐 | 评价、账户、风格、内容、多语言、数据看板全面增强 | 中 |

## 四、通用后台标准

### 1. 页面结构标准

每个后台列表页统一采用：

```text
页面标题区
├── 标题
├── 说明文案
└── 主操作按钮

数据概览区
├── 关键指标卡片
└── 状态分布

筛选区
├── 关键词搜索
├── 状态筛选
├── 时间范围
├── 高级筛选
└── 重置按钮

表格区
├── 批量选择
├── 排序
├── 分页
├── 状态标签
├── 行内操作
└── 空状态/错误状态

详情区
└── 抽屉或详情页
```

### 2. 表格标准

所有业务表格必须具备：
- 搜索
- 状态筛选
- 分页
- 排序
- 加载状态
- 空状态
- 错误状态
- 行内操作
- 高风险操作二次确认
- 操作结果反馈

### 3. 状态标签标准

订单、支付、库存、商品、评价、优惠券、活动都必须使用统一状态标签组件：

| 类型 | 示例 |
|------|------|
| 进行中 | pending、paid、shipped、refunding_payment、refunding |
| 成功 | completed、delivered、enabled、published |
| 警告 | low_stock、expiring、needs_review |
| 失败/终态 | cancelled、refunded、disabled、rejected |

### 4. 高风险操作标准

高风险操作包括：
- 发货
- 审批退款
- 拒绝退款
- 手动改订单状态
- 手动调库存
- 删除商品
- 禁用支付通道
- 修改支付配置
- 批量修改价格
- 批量上下架

统一要求：
1. 操作前展示影响范围。
2. 明确影响哪些表和数据。
3. 要求用户确认。
4. 后端再次校验权限和状态。
5. 写入审计日志。
6. 页面展示操作结果。

## 五、订单中心升级设计

### 1. 订单状态分组

| 后台分组 | 对应状态 | 说明 |
|----------|----------|------|
| 待付款 | pending | 用户已下单，未支付 |
| 待发货 | paid | 已支付，等待管理员发货 |
| 待收货 | shipped | 已发货，等待用户确认收货 |
| 退款审核 | refunding_payment | 管理员审核中 |
| 退款中 | refunding | 已发起支付平台退款 |
| 已退款 | refunded | 退款完成，资源已返还 |
| 已取消 | cancelled | 用户取消或超时取消 |
| 已完成 | completed | 交易完成 |

### 2. 订单详情页必须展示

- 订单基础信息
- 用户信息
- 收货地址
- 商品明细
- 原价、优惠、运费、最终金额
- 支付方式和支付状态
- 支付平台流水号 `reference_id`
- 优惠券使用明细
- 库存扣减/返还流水
- 订单状态时间线
- 后台操作日志

### 3. 订单操作规则

| 操作 | 允许状态 | 结果状态 | 附加动作 |
|------|----------|----------|----------|
| 发货 | paid | shipped | 记录发货信息 |
| 审批退款 | refunding_payment | refunding | 发起支付平台退款 |
| 拒绝退款 | refunding_payment | paid / shipped | 记录拒绝原因 |
| 查看支付流水 | 任意 | 不变 | 只读 |
| 查看库存流水 | 任意 | 不变 | 只读 |

### 4. 严禁行为

- 严禁后台按钮绕过状态机直接随意改状态。
- 严禁退款成功前归还库存和优惠券。
- 严禁重复发货。
- 严禁对非 pending 订单重新发起支付。

## 六、库存中心升级设计

### 1. 库存模块覆盖范围

- 库存总览
- 商品库存明细
- 库存流水
- 手动调库
- 入库
- 出库
- 损耗
- 退款返还
- 购物车预占库存
- 订单超时释放库存
- 库存盘点
- 低库存预警

### 2. 库存流水要求

每条库存流水必须包含：
- 商品 ID
- 商品名称
- 规格信息
- 变更前库存
- 变更数量
- 变更后库存
- `transaction_type`
- 关联业务单据
- 操作人
- 操作时间
- 备注

### 3. 字典约束

必须使用当前 `transaction_type` 字典，不允许页面硬编码业务类型。

重点类型：
- `order_cancel`：订单超时取消返还库存
- `refund_return`：退款完成返还库存
- `admin_adjust_increase`：后台加库存
- `admin_adjust_reduce`：后台减库存

## 八、支付中心升级设计

### 1. 支付中心模块

```text
支付中心
├── 支付通道
├── 支付订单
├── 退款订单
├── 回调日志
└── 异常处理
```

### 2. 支付通道管理

支持管理：
- PayPal
- Stripe
- Alipay
- 未来 WeChat

每个通道展示：
- 通道名称
- 启用状态
- 环境模式
- 币种支持
- 金额单位
- 最近成功支付时间
- 最近失败时间
- 异常数量

### 3. 支付订单管理

必须支持按以下字段查询：
- 订单号 `order_number`
- 支付平台流水号 `reference_id`
- 用户 ID
- 支付方式
- 支付状态
- 金额区间
- 创建时间
- 支付时间

### 4. 退款管理

退款必须遵循：

```text
refunding_payment → refunding → refunded
```

只有支付平台确认退款成功后，才允许进入 `refunded` 并触发库存和优惠券返还。

### 5. 支付安全底线

- 后台不得允许手动改订单金额。
- 支付接口不得接受前端金额。
- 支付中心只读展示支付结果，涉及退款必须走审批和支付平台接口。
- 回调日志必须保留验签结果和处理结果。

## 十五、测试与验收标准

### 1. 每阶段必跑

```bash
npm run build
npm run lint
```

### 2. 守护测试建议

新增后台守护测试，至少覆盖：
- 后台页面不得引用前台页面组件。
- 支付接口不得接受前端金额。
- 订单退款状态必须经过 `refunding_payment → refunding → refunded`。
- 库存返还只能发生在取消或退款完成节点。
- 高风险后台操作必须存在权限校验。
- 高风险后台操作必须写审计日志。

### 3. 页面级验收

每个模块验收：
- 页面能打开。
- 筛选能用。
- 分页能用。
- 状态标签正确。
- 操作按钮按状态显示。
- 错误状态友好。
- 加载状态统一。
- 不影响前台页面。

### 4. 业务级验收

核心链路验收：
- 创建订单后后台可见。
- 支付成功后订单进入待发货。
- 后台发货后订单进入待收货。
- 用户申请退款后后台可审批。
- 管理员同意退款后进入退款中。
- 支付平台退款成功后进入已退款。
- 已退款后库存和优惠券返还。
- 库存流水和订单日志一致。
