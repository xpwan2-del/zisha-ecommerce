# 紫砂电商项目开发技能

## 核心开发原则

### 1. 服务端逻辑优先

**必须遵守**：所有业务逻辑必须在服务端完成，前端只负责渲染展示。

**禁止事项**：

- ❌ 前端做价格计算
- ❌ 前端做库存判断
- ❌ 前端做活动状态判断
- ❌ 前端做任何业务逻辑

**正确做法**：

- ✅ API返回完整数据（包括计算后的价格、状态等）
- ✅ 前端直接渲染API返回的数据
- ✅ 所有判断逻辑在服务端完成

### 2. 不修改现有页面展示

**必须遵守**：只添加新功能，不修改现有页面布局和展示效果。

**禁止事项**：

- ❌ 修改现有CSS样式
- ❌ 修改现有组件结构
- ❌ 删除现有功能代码
- ❌ 替换现有组件

**正确做法**：

- ✅ 使用条件渲染添加新功能
- ✅ 新功能使用独立组件
- ✅ 保持现有布局不变
- ✅ 向后兼容旧数据

### 3. 先检查现有功能，避免重复开发

**必须遵守**：开发新功能前，先全面检查现有代码，确认没有可复用的功能后再创建新功能。

**检查清单**：

- [ ] 检查现有API列表（`src/app/api/` 目录）
- [ ] 检查现有组件（`src/components/` 目录）
- [ ] 检查现有工具函数（`src/lib/` 目录）
- [ ] 检查数据库表结构（`src/lib/db/` 目录）
- [ ] 检查现有页面（`src/app/` 目录）

**禁止事项**：

- ❌ 不检查现有功能直接创建新功能
- ❌ 重复实现已存在的API逻辑
- ❌ 重复创建相似的组件
- ❌ 忽略现有数据库表结构

**正确做法**：

- ✅ 开发前先列出所有现有相关功能
- ✅ 分析现有功能是否满足需求
- ✅ 如不满足，基于现有功能扩展
- ✅ 如完全不存在，才创建新功能
- ✅ 向用户报告检查结果和复用建议

### 4. 增量开发

**必须遵守**：新功能用新文件或新API实现，不替换或删除现有代码。

**禁止事项**：

- ❌ 直接修改现有API
- ❌ 删除现有数据库字段
- ❌ 替换现有组件
- ❌ 一次性修改多个功能
- ❌ 如果要做数据修改的时候不要直接操作

**正确做法**：

- ✅ 新功能创建新API端点
- ✅ 新功能创建新组件文件
- ✅ 数据库添加新字段（不删除旧的）
- ✅ 一个一个功能开发
- ✅ 需要和用户沟通后确认才可操作

## 代码注释规范

### 注释原则

**必须遵守**：所有业务逻辑代码必须有中文注释，解释"为什么这样做"而非"做了什么"。

**注释质量标准**：

- ✅ 注释要解释业务背景和原因
- ✅ 注释要说明非常规处理的考量
- ✅ 注释要标注已知限制或待优化点
- ❌ 不要注释显而易见的代码（如 `i++; // i自增`）
- ❌ 不要用英文注释（保持中文）

**正确示例**：

```typescript
// 业务逻辑：因为购物车合并时，如果原购物车商品已享受活动价，
// 重新计算会导致价格变化，所以需要先恢复原价再重新计算
if (promotionRule) {
  item.price = originalPrice;
}
```

### 注释模板

**函数注释**：

```typescript
/**
 * 函数名称：计算购物车最终价格
 * 功能说明：根据活动规则计算折扣后的价格
 * 业务规则：
 *   - 如果是会员日，全场95折
 *   - 如果有促销码，根据促销码类型计算
 *   - 如果同时有多个活动，按最高折扣计算
 * 输入参数：cartItems - 购物车商品列表, userId - 用户ID
 * 返回值：计算后的购物车总价
 * 注意事项：
 *   - 需要先调用getPromotions获取有效活动
 *   - 折扣计算采用"就高不就低"原则
 */
```

**复杂逻辑注释**：

```typescript
// 关键点说明：[为什么要这样做]
// 场景：用户同时有促销码和会员折扣
// 原方案：分别计算后叠加
// 问题：叠加后折扣低于预期，导致价格倒挂
// 最终方案：取最大折扣，不叠加
// 参考：促销活动规则文档-第三节-折扣互斥逻辑
```

**数据库操作注释**：

```sql
-- 操作：更新商品库存
-- 原因：用户下单成功，扣减库存
-- 并发考虑：使用乐观锁（version字段）防止超卖
-- 回滚处理：如果库存不足，返回错误码ERR_STOCK_INSUFFICIENT
UPDATE products
SET stock = stock - ?,
    version = version + 1
WHERE id = ? AND version = ? AND stock >= ?;
```

### 字段注释规范

**数据库表字段注释要求**：

| 字段类型   | 注释要求        | 示例                                                            |
| ------ | ----------- | ------------------------------------------------------------- |
| 关键业务字段 | 必须详细注释业务含义  | `total_price DECIMAL(10,2) COMMENT '最终成交价，含折扣，不含运费'`          |
| 状态字段   | 必须列出所有枚举值   | `status TINYINT COMMENT '订单状态：0-待付款，1-已付款，2-已发货，3-已完成，4-已取消'` |
| 关联字段   | 必须说明关联关系    | `user_id INT COMMENT '关联users.id，外键约束'`                       |
| 时间字段   | 必须说明时区和业务含义 | `created_at TIMESTAMP COMMENT '订单创建时间，北京时间，用户展示用'`            |
| 金额字段   | 必须说明币种和计算方式 | `discount_amount DECIMAL(10,2) COMMENT '折扣金额，人民币元，正数表示优惠'`    |

## 开发流程规范

### 开发前充分沟通

**必须执行**：任何功能开发前，必须先进行需求分析和方案设计，与用户充分沟通确认后再动手编码。

**沟通流程**：

```
用户发起需求
    ↓
AI分析需求，制定方案
    ↓
AI输出方案给用户确认（使用TodoWrite工具记录任务）
    ↓
用户确认方案
    ↓
AI才开始执行开发
    ↓
开发完成后，AI执行验证
    ↓
验证通过后，通知用户
```

**方案输出要求**：

每次开发前，AI必须向用户输出包含以下内容的方案说明：

1. **需求理解**：简述对需求的理解
2. **现有功能检查**：列出已检查的相关功能
3. **实现方案**：详细说明技术实现步骤
4. **影响范围**：说明会影响的文件和功能
5. **风险评估**：列出可能的风险点和应对措施
6. **开发计划**：使用 TodoWrite 工具列出具体步骤

**方案确认标志**：

用户回复"OK"、"确认"、"可以"等明确同意的词汇后，AI才开始开发。

如果用户提出修改意见，AI必须先修改方案，再次提交用户确认，直到用户明确同意。

**必须确认的场景**：

#### 阶段1：静态代码检查

**执行时机**：每次代码修改完成后立即执行

**执行命令**：

```bash
npx tsc --noEmit <修改的文件路径>
```

**通过标准**：

- 无TypeScript编译错误
- 无类型错误
- 无导入/导出错误

**失败处理**：

- 立即修复编译错误
- 重新执行静态检查
- 直到通过后才能继续

#### 阶段2：AI后端功能测试

**执行时机**：静态检查通过后执行

- 验证请求/响应格式正确
- 验证Cookie设置/清除/更新逻辑

**通过标准**：

- API返回正确状态码
- 响应JSON格式正确
- Cookie在响应头中正确设置
- 日志无500错误

### 1.4 折扣计算规则

**关键规则**：

1. **如果有独占促销（can\_stack=1）**：
   - 只取折扣**最高**的独占促销
   - 其他独占促销忽略
2. **如果没有独占促销（全可叠加）**：
   - 按优先级排序
   - 计算总体折扣率
   - 公式：`1 - (1-d1/100) × (1-d2/100) ×(1-d3/100) `  &#x20;

### 库存流水业务逻辑

#### 1. 扣库存场景（quantity\_change < 0）

| 场景    | API                            | 扣库存时机  |
| ----- | ------------------------------ | ------ |
| 加入购物车 | POST /api/cart                 | 立即扣库存  |
| 立即购买  | POST /api/inventory/reserve    | 跳转前扣库存 |
| 快速订单+ | PUT /api/quick-order/inventory | 立即扣库存  |

#### 2. 归还库存场景（quantity\_change > 0）

| 场景            | 触发条件     | 归还时机  |
| ------------- | -------- | ----- |
| 购物车-          | 数量减少     | 立即归还  |
| 购物车+          | 数量增加     | 立即扣库存 |
| 购物车删除         | 删除商品     | 立即归还  |
| 购物车清空         | 清空购物车    | 立即归还  |
| 快速订单页面订单商品数量- | 点击商品数量-号 | 立即归还  |
| 快速订单页面订单商品数量+ | 点击商品数量+号 | 立即扣库存 |
| 快速订单取消        | 取消订单     | 立即归还  |
| 退货完成          | 确认退货     | 立即归还  |

#### 3. 库存流水记录原则

1. **每一笔库存变动必须记录**：无论是扣库存还是归还库存
2. **记录必须包含**：transaction\_type\_id、quantity\_change、quantity\_before、quantity\_after
3. **流水不可删除**：历史流水只能新增，不能修改或删除
4. **余额必须一致**：quantity\_before + quantity\_change = quantity\_after

### 禁用逻辑核心代码

```typescript
const hasNonStackableSelected = selectedCouponIds.some(id => {
  const c = availableCoupons.find(ac => ac.id === id);
  return c && c.is_stackable === 0;
});

const hasStackableSelected = selectedCouponIds.some(id => {
  const c = availableCoupons.find(ac => ac.id === id);
  return c && c.is_stackable === 1;
});

const isDisabled = !isSelected && (
  coupon.is_stackable === 0
    ? hasStackableSelected || hasNonStackableSelected  // 不可叠加券：有其他券选中就禁用
    : hasNonStackableSelected                          // 可叠加券：有不可叠加券选中才禁用
);
```

### 优惠券过期判断逻辑

优惠券过期判断逻辑：
优惠券使用前必须客户先进领取优惠券
1. **is_active** | 1 |这个是核心中的核心 优惠券是否激活，1=激活可用，0=未激活 ，这里先要判断这个优惠券是否被激活在判断是否过期 这里过期有两个时间一个是优惠券的领取之前的优惠券时间（start_date，end_date），一个是用户领取后的过期时间（user_coupons.expires_at）换句话不管是客户领取了还是没领取只要is_active为1才可是使用或者被领取|
3. 客户可领取的优惠券是并且要判断 **coupon.start_date**  **coupon.end_date** **coupon.user_limit** | 2026-12-31 |  优惠券开始生效时间,优惠券失效时间，在这个时间周期内客户端客户看不到优惠券,并且客户如果之前领取过A优惠券且A优惠券是coupon.is_limit=1，那么客户就不能再领取A优惠券了,这里要去user_coupons表判断是否有A优惠券的记录|
4. 有有效期的优惠券客户的领取时间加上优惠券的有效期周期就是到期时间！**coupons.permanent_days** | 30 | 领取后有效天数，配合is_permanent使用，领取后30天内有效 |客户领取后更新user_coupons.expires_at 为领取时间加上permanent_days的时间 |
5. 即使客户领取了有有效期的优惠券还没有过有效期，但是如果优惠券本身没有被激活也不能使用**expires_at** | 2026-05-21 | 用户领取后的过期时间，超过此时间不可用 |还要判断coupon.is_active是否为1，1=激活可用，0=未激活 ，0=优惠券未激活，不可使用 |

这个是判断是否可领取优惠券
- 优惠券必须在生效时间内coupon.start_date <= datetime('now') <= coupon.end_date
- 优惠券必须coupon.is_active=1，1=激活可用，0=未激活 ，0=优惠券未激活，不可使用
- 优惠券是否只能领一次coupon.user_limit=1，1=只能领取1次，0=可以重复领取
