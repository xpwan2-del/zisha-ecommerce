## 二、审查问题分类

按以下 8 种类型对问题进行分类：

| 类别 | 说明 | 严重程度定义 |
|-----|------|-------------|
| 代码规范 | TypeScript/React 最佳实践、命名规范 | 低/中/高 |
| 安全漏洞 | SQL注入、XSS、CSRF、认证授权问题 | 高/中/低 |
| 逻辑错误 | 业务逻辑缺陷、状态处理错误 | 高/中/低 |
| 性能问题 | N+1查询、内存泄漏、不必要重渲染 | 高/中/低 |
| 可维护性 | 代码重复、耦合度过高、缺少注释 | 低/中 |
| API设计 | 响应格式不一致、错误处理不规范 | 高/中/低 |
| 状态管理 | Context使用不当、全局状态滥用 | 中/低 |
| 数据库设计 | 字段设计不合理、索引缺失 | 高/中/低 |

## 三、严重程度定义

### 3.1 严重（High）
- **定义**：影响核心业务流程、导致数据丢失或安全风险
- **处理**：必须修复后才能上线

### 3.2 中等（Medium）
- **定义**：影响用户体验、存在潜在风险但有临时方案
- **处理**：建议本次迭代修复或下个迭代修复

### 3.3 低（Low）
- **定义**：代码风格、命名优化等非功能性改进
- **处理**：可根据情况推迟

## 五、审查规范遵循检查

根据 `API-DEV-GUIDE.md` 规范，检查：

### 5.1 API 文件结构
- [ ] 是否包含 OpenAPI 注释（@api, @apiName, @apiGroup, @apiDescription）
- [ ] 是否包含 @apiHeader, @apiParam, @apiSuccess, @apiError 注释
- [ ] 是否包含成功/错误响应示例

### 5.2 监听框架接入
- [ ] 是否使用 logMonitor 进行请求监听
- [ ] 模块归类是否正确（AUTH/CART/ORDERS/INVENTORY/PAYMENTS/PRODUCTS/API）
- [ ] 操作类型是否正确（REQUEST/SUCCESS/ERROR/VALIDATION_FAILED/AUTH_FAILED/NOT_FOUND）

### 5.3 响应格式
- [ ] 是否使用统一的成功响应格式：`{ success: true, data: {...} }`
- [ ] 是否使用统一的错误响应格式：`{ success: false, error: 'ERROR_CODE' }`
- [ ] HTTP 状态码使用是否正确

### 5.4 多语言支持
- [ ] 是否从请求头 x-lang 或 cookie 获取语言设置
- [ ] 错误消息是否支持多语言

### 5.5 错误处理
- [ ] 是否使用 messages.ts 中的错误码
- [ ] 是否避免硬编码错误信息
- [ ] 是否正确处理 try-catch
