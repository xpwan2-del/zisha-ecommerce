#  紫砂电商项目 全面分析报告 分析日期：2026-05-02 | 项目：zisha-ecommerce | 分析范围：39张表、91个API、33个页面、30个组件、34个后端工具文件
## 一、项目总体概况
维度 数量 状态 数据库表 39 张 丰富但存在架构问题 API 接口文件 91 个 route.ts 量大但规范不统一 前端页面 33 个 (含 2 layout) 前台14 + 后台17 前端组件 30 个 覆盖核心场景 后端工具文件 34 个 架构较完善 支付通道 Paypal / Alipay / Stripe 三通道但实现程度不一 多语言 zh / en / ar 全覆盖但翻译质量参差

## 二、🔴 严重问题（P0 — 必须立即修复）
### 2.1 数据库事务完全失效
问题 ： db.ts 中 BEGIN TRANSACTION 、 COMMIT 、 ROLLBACK 被静默忽略为 NO-OP。但业务代码中 11 个文件 仍在使用事务语句。

```
后果：
- ROLLBACK 不会回滚任何数据
- 订单创建到一半失败，已插入的数据仍然持久化
- 开发者获得了虚假的安全感
```
影响范围 ： cart/create-order 、 orders/route.ts 、 order-status-service.ts 等所有依赖事务的核心流程。

### 2.2 SQL 注入漏洞（高危）
问题 ： user/profile/route.ts 中直接将用户可控的字段名拼入 SQL：

```
const oldResult = await query(`SELECT ${field} FROM users WHERE id = ?`, [user_id]);
```
field 来自 request.json() 的 key，攻击者可构造任意字段名注入恶意 SQL。

### 2.3 库存扣减存在超卖风险
问题 ：由于事务不工作，"SELECT 查库存 → 判断 → UPDATE 减库存" 之间存在 TOCTOU 时间窗口。

此外， quick-order/create/route.ts 的快速订单确认流程 完全没有库存扣减逻辑 ——既不扣库存也不检查库存。

### 2.4 两套订单创建流程使用不同的数据库列名
路径 金额字段 /api/orders POST total_amount /api/cart/create-order POST total_after_promotions_amount + total_original_price + total_coupon_discount

两套流程使用不同字段导致数据不一致和查询时出现 NULL 值。

## 三、🟠 重大问题（P1 — 短期必须修复）
### 3.1 价格双源不一致
系统存在两套价格存储：

- products 表 ：直接存储 price (CNY)、 price_usd 、 price_ae
- product_prices 表 ：以 (product_id, currency) 组合存储
不同 API 从不同来源读取，管理后台修改价格可能不同步到 product_prices 表，导致前端列表和管理后台显示不同价格。

### 3.2 优惠券限制条件完全无效
优惠券表有 min_spend （最低消费）和 max_discount （最大折扣上限）字段，但所有优惠券验证逻辑 完全不检查 这两个字段。一张设置了最低消费 500 元的优惠券可以被用于 10 元订单。

### 3.3 忘记密码是死链接
login/page.tsx 中"忘记密码"链接指向 href="#" ，用户点击无任何反应。缺少：

- /api/auth/forgot-password 发送重置邮件的 API
- /api/auth/reset-password 重置密码的 API
- 密码重置 Token 生成/验证机制
### 3.4 订单取消有配置但无用户 API
项目中定义了完整的取消状态转换规则（ user_cancel 、 timeout_cancel 、 admin_cancel ），支付通道的 cancel 路由也已实现。但 用户无法主动取消订单 ——缺少用户订单取消 API 端点。

### 3.5 退货退款有配置但无 API
状态机已定义 refunding / refunded 状态和 refund_request / refund_success / refund_reject 事件，但 没有任何退货退款 API 实现 。

## 四、🟡 中等缺失（P2 — 影响业务完整性）
### 4.1 邮件通知完全缺失
项目中 没有任何邮件发送代码 （无 nodemailer、无 SendGrid、无 SMTP 调用）。以下场景均无通知：

- 注册成功欢迎邮件
- 密码重置邮件
- 订单确认邮件
- 订单发货通知
- 支付成功/失败通知
### 4.2 未支付订单库存不会自动释放
inventory/reserve/route.ts 中的注释明确写道：「如果用户放弃支付，库存不会自动释放（需要定时任务清理）」。但 没有定时任务 。

### 4.3 订单管理后台缺失
管理后台唯一 没有 的页面是订单管理。管理员无法在后台：

- 查看所有订单列表
- 修改订单状态（发货、退款等）
- 管理物流信息
### 4.4 注册无需邮箱验证
注册后立即登录，无邮箱验证、无验证码。 users 表没有 email_verified 字段。

### 4.5 API 无速率限制
middleware.ts 仅做日志记录，无任何速率限制。登录 API 可被暴力破解。

### 4.6 运费被硬编码为 0
三处订单创建代码中，两处将运费硬编码为 0：

- cart/create-order ： const shippingFee = 0;
- orders/route.ts ： const shipping_fee = 0; // Free shipping for now
### 4.7 汇率硬编码在代码中
quick-order/create/route.ts AED→USD 汇率硬编码为 0.2722，AED→CNY 汇率硬编码为 1.9558。汇率变更需要修改代码并重新部署。

## 十、结论
该项目 基础架构扎实 ，具备一个完整商城的基本骨架（商品、购物车、订单、支付、库存、多语言），代码量和功能覆盖已经相当丰富。

但存在 4 个 P0 级严重问题 （事务失效、SQL 注入、库存超卖、列名不一致）需要在任何上线前立即修复。另有 7 个 P1 重大问题 影响核心业务流程完整性（忘记密码、取消订单、退货退款、库存自动释放等）。此外， 10 项标准商城功能完全缺失 ，在中东市场运营中尤为重要的 WhatsApp 分享、货到付款、客服聊天等功能需要补充。

建议按 Phase 1 → 2 → 3 → 4 → 5 的顺序逐步推进修复和完善。
